Kolmogorov Complexity: What Does "Simple" Mean?

Which of these two 32-character strings is more "random": 01010101010101010101010101010101, or 10011010001101110010100111001011? Gut instinct says the second — the first has an obvious pattern, the second doesn't. This lesson makes that gut instinct precise, and then does something more interesting: uses the same self-reference trick from Logic lesson 15 to prove that the precise version of "simple" can never be fully computed.

The definition

Kolmogorov Complexity Fix some universal programming language. The Kolmogorov complexity of a string x, written K(x), is the length (in bits) of the shortest program in that language that outputs x and halts.

The repeating string above has low K — a program like print "01" 16 times is short, far shorter than the string itself. The second string, if it really has no exploitable pattern, needs a program that's basically print "100110100011011100101001110010" + "11" — i.e., the shortest program is roughly the string itself, plus a little overhead for print. That's the whole intuition: K(x) is a rigorous stand-in for "how compressible is x."

Doesn't this depend on which language you picked? A little, but not in a way that matters asymptotically. The invariance theorem says: for any two universal languages L₁ and L₂, there's a constant c (the length of an L₁-to-L₂ interpreter, roughly) such that |K_L₁(x) − K_L₂(x)| ≤ c for every x. As strings get longer, this fixed overhead becomes negligible — so "the shortest program" is a well-defined notion up to a constant that doesn't depend on x, which is exactly the sense in which K is language-independent.

Algorithmic randomness — when no shortcut exists

Call a string x of length n algorithmically random (or incompressible) if K(x) ≥ n − c for some small constant c — no significantly shorter description exists than just writing x out in full. A counting argument shows something that sounds strange at first: most strings of any given length are incompressible. There are 2ⁿ strings of length n, but only 2ⁿ⁻¹ − 1 possible programs shorter than n−1 bits — nowhere near enough short programs to give every long string a short description. Pigeonhole forces most strings to have no shortcut at all.

The strange part isn't the theorem — it's what it implies about specific strings. You can prove almost all strings are incompressible, but (as the next section shows) you can essentially never prove a specific given string is one of them. Randomness, on this definition, is common but individually hard to certify.

Why K is uncomputable — the same move as Gödel

Theorem: K is not computable There is no program that, given any string x, always outputs K(x).

The proof is a direct cousin of lesson 15's diagonal lemma — a self-reference trap dressed up as an old riddle called the Berry paradox ("the smallest number not describable in fewer than twelve words," which that very phrase just described in eleven).

  • Suppose, for contradiction, a program K-compute exists that always outputs K(x) given x, and suppose it's P bits long.
  • Write a new program find-first-complex(N): search all strings in order and, using K-compute, output the first string x such that K(x) ≥ N. Such a string is guaranteed to exist by the counting argument above, for any N once strings are long enough.
  • find-first-complex's own length is roughly P + log(N) + (a small constant) — the cost of K-compute, plus writing the number N, plus a little bookkeeping. Crucially, this grows only logarithmically in N, while N itself grows linearly.
  • So for N large enough, find-first-complex(N) is a program shorter than N bits that outputs a string whose K is at least N — meaning that string has a description (this very program) shorter than N bits, so its real K is less than N. Contradiction: the program's own output has both K(x) ≥ N (by construction) and K(x) < N (because we just described it briefly). K-compute cannot have existed.
Recognize the shape This is the exact same family as Gödel's G ("this sentence is unprovable") and the halting problem ("this program halts iff it doesn't"): construct an object using a hypothetical solver, engineered specifically to contradict what that solver would say about it. Three different fields — logic, computability, information theory — and the same one proof technique breaks all three the same way.

Not the same "information" as Shannon entropy

Classical (Shannon) information theory asks: given a known probability distribution over messages, what's the average number of bits needed to encode a message drawn from it? That's a statement about a source, not any single message. Kolmogorov complexity asks a different question entirely: given one specific string, with no reference to any distribution it was drawn from, how short can its shortest description be? A single, highly patterned string can have low K even if it was drawn from a distribution with high entropy (you just got a low-probability, highly-compressible outcome). The two "information" concepts agree on average, in a precise sense, but they are not interchangeable, and conflating them is a common, real error.

Practice before next lesson

Take three strings of your own choosing, all the same length (say 24 characters): one obviously patterned (like a repeating sequence), one you generate by actually flipping a coin 24 times, and one that looks random to you but that you made up "randomly" off the top of your head. Write, in plain English, what the shortest program you can think of for each would look like. Then consider: which one do you expect to be hardest to actually prove is incompressible, even if it probably is? (Hint: it's not the coin-flip one you'd guess.)
