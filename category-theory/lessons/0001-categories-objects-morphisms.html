Categories, Objects & Morphisms

You already know that functions compose: given f: A → B and g: B → C, you can build g∘f: A → C, and it doesn't matter whether you compute (g∘f)∘h or g∘(f∘h) first. Category theory asks: what's the most general setting in which "things that compose, associatively, with a do-nothing identity" makes sense at all? The answer turns out to describe function composition, sets, logical proofs, and a great deal else, all at once — which is exactly the point.

A category, precisely

Definition: Category A category consists of:
  • Objects — think of these as "the things," whatever they are (they don't need internal structure at all)
  • Morphisms (arrows) — for every ordered pair of objects A, B, a set of arrows f: A → B
  • Identity — every object A has an identity morphism id_A: A → A
  • Composition — arrows f: A → B and g: B → C compose to give g∘f: A → C
subject to two laws:
  • Identity law: f ∘ id_A = f and id_B ∘ f = f — composing with identity changes nothing
  • Associativity: (h∘g)∘f = h∘(g∘f) — grouping doesn't matter

Notice what's not in this definition: nothing about what objects "are" internally, nothing about what morphisms "do" beyond composing correctly. A category is defined entirely by how its arrows compose — which is deliberate, and is most of the point.

Three genuinely different categories

The temptation is to think "objects = sets, morphisms = functions" and call it a day. That's one category (called Set), but the definition doesn't require anything like it:

  • Set — objects are sets, morphisms are functions between them, composition is ordinary function composition. The familiar case.
  • Your own type system — objects are types (i32, String, a struct), morphisms A → B are functions with that signature, identity is the identity function, composition is ordinary function composition — but now "object" means type, a compile-time concept, not a runtime value.
  • A preorder — pick any set with a "≤"-like relation (say, natural numbers under ≤). Objects are the numbers themselves. There is a morphism m → n exactly when m ≤ n — and never more than one. Composition is transitivity (m≤n and n≤p give m≤p); identity is reflexivity (m≤m). There's no "function" here at all — a morphism is just the bare fact that the relation holds.
Why the preorder example matters It's the cleanest proof that a category isn't secretly "just functions with extra steps." In a preorder-as-category, morphisms aren't computations you run — they're witnesses to a relationship being true. Composition isn't "do this, then that" — it's "if this holds and that holds, then the combined thing holds too." Same axioms, wildly different intuition. That gap is exactly what makes the abstraction useful: any proof you do purely from the category axioms works in both settings, and everywhere else the axioms happen to hold, automatically.

Checking the laws on a tiny example

Take a category with exactly two objects, A and B, and exactly one non-identity morphism f: A → B (plus the two required identities, id_A and id_B). Check the identity law by hand: f ∘ id_A must equal f — there's genuinely only one arrow A → B available for it to equal, so this holds trivially here. Associativity has nothing to check in this tiny example (there's no triple of composable non-identity arrows), which is itself a useful lesson: the axioms need to hold for every valid composition in a category, but a specific tiny category can satisfy them by simply not having enough arrows to violate them. This is why the counterexamples worth building are ones with more structure — enough arrows that a law could plausibly fail, and then checking whether it actually does.

Practice before next lesson

Pick a small Rust or C type signature you actually use — say, three functions f: A → B, g: B → C, h: C → D from real or invented types. Write out what h∘(g∘f) and (h∘g)∘f would each compute, step by step, and confirm they give the same function. Then name one identity function for one of your types and confirm composing it with f changes nothing. You're checking, by hand, that your own familiar type system already satisfies both category laws — which it does, always, for any programming language's types and functions.
