Functors & Natural Transformations

Lesson 1 built categories — objects and arrows within one structure. This lesson does what category theory always does next: go up one level. First, a map between categories that respects their structure (a functor). Then, a map between two such maps (a natural transformation). If you've written .map() on an Option or a Vec, you've already used a functor — this lesson names what you were doing and shows you the law it was quietly obeying the whole time.

A functor, precisely

Definition: Functor A functor F: C → D from category C to category D assigns:
  • to every object A in C, an object F(A) in D
  • to every morphism f: A → B in C, a morphism F(f): F(A) → F(B) in D
subject to two laws:
  • Preserves identity: F(id_A) = id_F(A)
  • Preserves composition: F(g ∘ f) = F(g) ∘ F(f)

In plain terms: a functor doesn't just relabel objects — it has to drag every arrow along consistently, in a way that respects how those arrows composed in the first place. That "in a way that respects composition" clause is doing all the real work; without it, you could map objects and morphisms around arbitrarily and call it a functor, and the whole idea would be useless.

The functor you already use: Option::map

Take C = D = your own type system (lesson 1's second example). Define F by: on objects, F(T) = Option<T> — every type T maps to the type "maybe a T." On morphisms, given f: A → B, define F(f) = Option::map(f) : Option<A> → Option<B> — apply f inside if there's a value, pass None through untouched. Check the laws:

  • Identity: Option::map(id) applied to Some(x) gives Some(id(x)) = Some(x); applied to None gives None. Either way, unchanged — exactly id on Option<A>. ✓
  • Composition: Option::map(g∘f) applied to Some(x) gives Some(g(f(x))). Doing it in two steps — Option::map(g) applied to Option::map(f) applied to Some(x) — gives Option::map(g) on Some(f(x)), which is Some(g(f(x))). Same result either way. ✓
Why the laws matter in practice These two laws are exactly what makes .map() "safe" to refactor without thinking twice: you can always merge two consecutive .map(f).map(g) calls into one .map(|x| g(f(x))), or split them apart, and the compiler-verified behavior never changes. That's not a coincidence or a convention — it's the composition law, holding in your actual code every time you've relied on this kind of refactor being safe.

A functor that isn't a container at all

Functors aren't only about "boxes with a value inside." Recall lesson 1's preorder category (numbers under ≤, a morphism m→n whenever m≤n). Define F from that preorder into Set: send every number n to the set {1, 2, ..., n}, and send the morphism "m≤n" to the inclusion function embedding {1,...,m} into {1,...,n}. This is a completely legitimate functor — it preserves identity (m≤m maps to the identity inclusion) and composition (m≤n≤p's combined inclusion is the same as composing the two smaller inclusions) — and there's no "container holding a value" anywhere in sight. The lesson: a functor is defined by what it does to objects and arrows, full stop; "generic container type" is one common shape functors take, not what a functor is.

Natural transformations — a map between functors

Definition: Natural Transformation Given two functors F, G : C → D, a natural transformation η : F ⇒ G assigns to every object A in C a morphism η_A : F(A) → G(A) in D, such that for every morphism f: A → B in C, this naturality square commutes: 
G(f) ∘ η_A  =  η_B ∘ F(f)
— i.e., it doesn't matter whether you convert first (η) then map with G, or map with F first then convert (η) — same result either way.

Concrete example: let F be the Option functor from above, and G be the Vec functor (G(T) = Vec<T>, morphisms lifted via ordinary list .map()). Define η_T : Option<T> → Vec<T> by None ↦ [], Some(x) ↦ [x] — the same conversion rule, for every type T. Check naturality for some f: A → B: starting from Some(x), converting-then-mapping gives Vec::map(f)([x]) = [f(x)]; mapping-then-converting gives η(Some(f(x))) = [f(x)]. Same result. Starting from None, both paths give []. The square commutes for every f, every type — which is exactly the sense in which "turn a maybe-value into a maybe-empty list" is the same operation regardless of what's inside, not a family of unrelated conversions that happen to look similar.

What naturality is actually ruling out Imagine instead defining a "conversion" that behaves differently depending on the type T — say, one that works fine for Option<i32> but does something special-cased for Option<String>. That could still be a legal function type-by-type, but it would fail naturality: the square wouldn't commute for a morphism f connecting those types, because the "conversion" isn't actually one uniform operation — it's secretly several different ones wearing the same name. Naturality is the formal way of saying "this transformation doesn't secretly peek at which type it's running on."

Practice before next lesson

Define your own tiny functor: pick a simple container type (say, a "box" type holding exactly one value, or a pair type holding two values of the same type) and write out, in words or pseudocode, what its map function does. Then check both functor laws by hand on a couple of concrete functions, the way this lesson checked them for Option. Finally, try defining a natural transformation from your box/pair functor to Vec, and check the naturality square holds for at least one function f — the goal is to build one of these yourself instead of only verifying the lesson's examples.
