Monads

There's a famous joke: "a monad is just a monoid in the category of endofunctors." By the end of this lesson that sentence will be a literal, checkable description, not a punchline — and you'll recognize the pattern as something you've already used every time you've chained Option or Result operations in Rust.

The problem: composing functions that don't quite compose

Lesson 2's functor let you lift an ordinary function f: A → B to Option::map(f): Option<A> → Option<B>. But what about a function that's itself fallible — f: A → Option<B>? Given a second such function g: B → Option<C>, ordinary composition g∘f doesn't typecheck: f's output is Option<B>, but g expects a plain B. You need a principled way to chain functions whose outputs already carry extra context (maybe-absent, error-or-success, many-possible-results) — and "principled" here means: a general recipe that works for any such context, not a one-off fix per type. That recipe is a monad.

The definition

Definition: Monad A monad on category C is an endofunctor T: C → C (a functor from C to itself) together with two natural transformations:
  • unit η: Id ⇒ T — for every object A, a morphism η_A: A → T(A) (wrap a plain value in the context)
  • join μ: T∘T ⇒ T — for every object A, a morphism μ_A: T(T(A)) → T(A) (collapse two layers of context into one)
satisfying laws directly analogous to a monoid's associativity and identity axioms (made precise below).

The Option monad, concretely

T(A) = Option<A> is already a functor (lesson 2). The monad structure adds:

  • unit: η_A(a) = Some(a) — wrap any plain value.
  • join: μ_A: Option<Option<A>> → Option<A> — flatten: None ↦ None, Some(None) ↦ None, Some(Some(a)) ↦ Some(a).

Check one instance of the identity law by hand: μ_A(η_{Option<A>}(Some(x))) — apply η at type Option<A> to get Some(Some(x)), then μ flattens it to Some(x). Unchanged, as the law requires. This particular combination of unit-then-join undoing itself, for every object, isn't a coincidence for Option specifically — it's exactly what the monad laws demand of any valid monad.

Where "bind" (and_then, ?) comes from

The operation you actually reach for day to day — Rust's .and_then(f), or the ? operator's short-circuiting — is bind, and it's not a separate primitive; it's unit and join in disguise: given m: T(A) and f: A → T(B), define bind(m,f) = μ(T(f)(m)) — apply the functor's own lifted f (giving T(T(B)), a value nested two levels deep), then flatten with μ. For Option: bind(Some(x), f) = μ(Some(f(x))) = f(x), and bind(None, f) = μ(None) = None — exactly the short-circuiting behavior .and_then() already gives you. Chaining bind(bind(m, f), g) for f: A→Option<B>, g: B→Option<C> now composes cleanly, automatically propagating None through every step — precisely the composition problem from the top of this lesson, solved once, generally, for any monad.

Cashing in the joke

"A monoid in the category of endofunctors" — literally A monoid is a set M with an associative binary operation and an identity element. Now build a category whose objects are endofunctors C → C and whose morphisms are natural transformations between them. In this category, functor composition plays the role of the binary operation, and the identity functor plays the role of the identity element. A monad — an endofunctor T with μ: T∘T ⇒ T and η: Id ⇒ T — is exactly a monoid object living inside that category: μ is the "multiplication" (combining two applications of T into one, the way a monoid combines two elements into one), η is the "identity element" (inserting a value with no real effect, the way a monoid's identity element changes nothing when combined). The monad laws (associativity: μ∘(μT) = μ∘(Tμ); identity: μ∘ηT = id = μ∘Tη) are, symbol for symbol, the monoid axioms, just stated in terms of natural transformations between endofunctors instead of elements of a set. The "joke" was always a precise, checkable claim — it just assumed you already knew what a monoid, an endofunctor, and a natural transformation were, which by this lesson, you do.

Practice before next lesson

Write out, by hand or in pseudocode, three functions f: A→Option<B>, g: B→Option<C>, h: C→Option<D> of your own invention (they don't need to be real code — just plausible fallible operations, like "parse a string," "look up a key," "divide two numbers").
