Adjunctions

You've already used two adjunctions without a name for them: currying a two-argument Rust function into a function that returns a function, and building a Vec<T> from raw elements and then .fold()-ing it back down. Adjunctions are the single idea that explains why both of those "just work," why lesson 3's universal properties keep showing up in pairs, and — as the closing callout below shows — where lesson 4's monads actually come from.

The pattern: a "best approximate" round trip

A functor F: C → D and a functor going the other way, G: D → C, rarely undo each other exactly — G(F(A)) usually isn't literally A. But in a huge number of important cases, there's still a precise, checkable relationship between them: mapping out of F(A) in D is exactly the same information as mapping out of A into G(B) in C, for every A and B. That relationship — not equality of the functors, but a controlled, natural correspondence between the morphisms they produce — is an adjunction, and F is said to be left adjoint to G (written F ⊣ G, with G the right adjoint).

The definition

Definition: Adjunction Functors F: C → D and G: D → C form an adjunction F ⊣ G when there is a bijection

Hom_D(F(A), B)  ≅  Hom_C(A, G(B))

for every object A in C and B in D, and this bijection is natural in both A and B — it doesn't secretly special-case which objects you plugged in, the same requirement lesson 2's naturality square imposed on natural transformations.

Read left to right: a morphism out of F(A) in D is freely interchangeable with a morphism into G(B) from A back in C. Nothing here says F and G are inverses — G(F(A)) is generally a bigger, more structured object than A. The adjunction only promises that morphisms on either side carry the same information.

Unit and counit

The hom-set bijection has an equivalent, more operational formulation: two natural transformations,

  • unit η: Id_C ⇒ G∘F — for every A, a morphism η_A: A → G(F(A))
  • counit ε: F∘G ⇒ Id_D — for every B, a morphism ε_B: F(G(B)) → B

satisfying the triangle identities (ε∘F)∘(F∘η) = id_F and (G∘ε)∘(η∘G) = id_G — the categorical way of saying the unit and counit don't fight each other; going "out and back" via either functor first is a no-op on the other. This is the same shape as lesson 4's monad unit, but now between two different categories via two different functors, instead of one endofunctor talking to itself. You'll see exactly how they connect in the callout near the end of this lesson.

Worked example 1: currying is an adjunction

Fix a set (or type) B. Define F(A) = A×B and G(C) = C^B — read C^B as "the set of functions B → C," i.e. Rust's impl Fn(B) -> C. The claim: F ⊣ G, via

Hom(A×B, C)  ≅  Hom(A, C^B)

which is exactly curry/uncurry in Rust: a function fn(A, B) -> C corresponds exactly to a function fn(A) -> (fn(B) -> C), with no information gained or lost either way. Under unit/counit:

  • unit η_A: A → (A×B)^B, η_A(a) = λb.(a, b) — pair a with whatever B-value shows up later.
  • counit ε_C: C^B×B → C, ε_C(f, b) = f(b) — this one is worth sitting with: the counit of the currying adjunction is literally function application. Every time you call a closure, you're using this counit.

Worked example 2: free monoid, forgetful functor, and fold

Let U: Monoid → Set be the forgetful functor — take a monoid and just remember its underlying set, throwing away the operation. Let F: Set → Monoid be the free functor — take a set A and build List<A>, with concatenation as the operation and the empty list as identity. This is lesson 3's "the smallest thing satisfying the property" move: List<A> is the monoid you get by adding only what the monoid axioms force and nothing else. The claim: F ⊣ U, via

Hom_Monoid(List<A>, M)  ≅  Hom_Set(A, U(M))

Right to left: given a plain function f: A → U(M) — no monoid structure required, just "for each element of A, pick a value in M" — there's exactly one monoid homomorphism List<A> → M agreeing with it, and you already know its name: fold. Concretely, ĥ([a₁,...,aₙ]) = f(a₁) ⊕ f(a₂) ⊕ ... ⊕ f(aₙ) (empty list ↦ M's identity), which in Rust is xs.into_iter().map(f).fold(M::identity(), M::combine). The adjunction is the formal statement that this is the unique homomorphism extending f — the same uniqueness clause that made lesson 3's universal properties bite.

Where lesson 4's monads actually come from Every adjunction F ⊣ G produces a monad on C: take T = G∘F as the endofunctor, the adjunction's own η as the monad's unit, and μ = G∘ε∘F (apply the counit "in the middle," then let G carry it back) as join. The Option monad from lesson 4 isn't a special case invented from nowhere — it's G∘F for the free/forgetful adjunction between Set and pointed sets (a set with one designated basepoint, plus basepoint-preserving functions): F(A) = A ⊔ {*} adds a fresh basepoint to A (this is Option<A> — the basepoint is None, everything else is wrapped in Some), and U forgets which point was distinguished. T = U∘F sends A to Option<A>, exactly recovering lesson 4's monad. The "monoid in the category of endofunctors" joke had a second layer underneath it the whole time.

Practice before next lesson

Take the free monoid adjunction and make it concrete: let A = {x, y} and let M be the monoid of integers under addition (identity 0). Pick any plain function f: A → ℤ — say f(x) = 3, f(y) = -1 — and hand-compute what the unique extended homomorphism ĥ: List<A> → ℤ sends the list [x, y, x, x] to. Then separately verify the currying adjunction on a toy example: write (in pseudocode) a two-argument function f(a, b), curry it into g(a)(b), and confirm g(a)(b) = f(a, b) for a couple of concrete inputs — that equality, holding for every a and b, is precisely the naturality of worked example 1's bijection.
