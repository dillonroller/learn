Big-O & Complexity

Every interview question you'll ever face has a hidden subtext: "Can you tell me how efficient your solution is, and why?" Big-O notation is the language for that conversation. You cannot pass a Google-level screen without it.

What Big-O actually measures

Big-O describes how the runtime or memory usage of an algorithm grows as the input size grows. It intentionally ignores constants and small inputs — it only cares about the dominant trend at scale.

Two key dimensions, and interviewers will ask about both:

  • Time complexity — how many operations does it take?
  • Space complexity — how much extra memory does it use?
Why interviewers care A brute-force O(n²) solution on a 10-element array is fine. On a 10-million-element array, it does 100 trillion operations and your server dies. Interviewers aren't being pedantic — they're testing whether you think about production scale.

Reading complexity from code

You need to be able to look at code and state its complexity immediately. Here are the patterns:

O(1) — no loops, no recursion depending on input

fn first(v: &[i32]) -> Option<i32> {
    v.get(0).copied()   // one operation regardless of v.len()
}

O(n) — one loop over the input

fn sum(v: &[i32]) -> i32 {
    v.iter().sum()   // visits every element once
}

O(n²) — nested loops over the same input

fn has_duplicate(v: &[i32]) -> bool {
    for i in 0..v.len() {
        for j in (i + 1)..v.len() {   // inner loop grows with n
            if v[i] == v[j] { return true; }
        }
    }
    false
}
// O(n²) time, O(1) space — an interviewer will immediately ask:
// "Can you do better?" (Yes: use a HashSet → O(n) time, O(n) space)

O(log n) — halving the problem each step

fn binary_search(v: &[i32], target: i32) -> Option<usize> {
    let (mut lo, mut hi) = (0, v.len());
    while lo < hi {
        let mid = lo + (hi - lo) / 2;   // halve the search space
        match v[mid].cmp(&target) {
            std::cmp::Ordering::Equal   => return Some(mid),
            std::cmp::Ordering::Less    => lo = mid + 1,
            std::cmp::Ordering::Greater => hi = mid,
        }
    }
    None
}
Common interview trap Space complexity counts extra memory your algorithm allocates, not the input itself. An algorithm that copies the input into a new array is O(n) space even if the logic is O(1) operations.

How to talk about it in an interview

Don't wait to be asked. After you sketch a solution, immediately say:

"This is O(n) time and O(1) space. If we need to reduce space further we could… but that would increase time to O(n log n)."

This signals that you think about trade-offs — exactly what senior engineers do.

Drop the constants — but understand why

O(3n) simplifies to O(n). O(n + n²) simplifies to O(n²). We drop lower-order terms and constants because at large scale they stop mattering. But in a real interview, if you have two O(n) solutions and one has a constant 3x overhead, it's worth mentioning — just don't let it change your Big-O label.
